Decidere di non creare una classe per ogni singolo tipo di edificio (es. class Condominio extends Residenza, class Villa extends Residenza), che varierebbero solo di valori, ma usare un'unica classe e variarne solo i dati interni:
1. evita abuso di polimorfismo:
    Cosa violeresti: Se creassi 10 classi diverse di case in cui l'unica differenza è che una ha capacita = 5 e l'altra capacita = 20, staresti abusando dell'ereditarietà.
        Il comportamento "far abitare le persone e riscuotere l'affitto" è identico per entrambe!
    Cosa rispetti: Usando un'unica classe e cambiando i valori dei componenti, dimostri di aver capito che l'ereditarietà
        serve per le variazioni di logica, salvando il sistema da decine di file .java inutili.

2. Type Object Pattern
Il pattern dice: Se hai molti "tipi" di un oggetto che condividono la stessa logica ma hanno statistiche diverse,
    non usare le classi del linguaggio di programmazione (Java) per rappresentare i "tipi".
    Usa una singola classe e passale un oggetto "Tipo" come parametro.
